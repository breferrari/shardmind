### Added (pre-release smoke gate — #112)

## [0.1.2] - 2026-04-26

### Fixed (iterated diff prompts freeze after the first decision)

Closes [#109](https://github.com/breferrari/shardmind/issues/109). Hit during a real `shardmind adopt github:breferrari/obsidian-mind` run on the flagship right after #103's 0.1.1 fix unblocked the install wizard: 35 differing files, "Keep mine" worked on file 1, every prompt thereafter was frozen.

- **`source/components/AdoptDiffView.tsx`** + **`source/components/DiffView.tsx`** — both held a boolean `firedRef = useRef(false)` to dedupe `Select.onChange` within a single mount. The parents (`adopt.tsx`, `update.tsx`) advance `phase.currentIndex` via the state machine without passing a `key` prop, so React kept the same component instance, `useRef` returned the same object, and `firedRef.current` stayed `true` from the previous file's resolution. Replace with a per-iteration scoped guard (see hook below); when the path changes, the next file's first Enter goes through. No parent-side `key` prop required — the encapsulation lives with the component.
- **`source/components/use-once-per-key.ts`** (new) — extracts the per-iteration dedup pattern into `useOncePerKey<K>(currentKey: K): () => boolean`. Both diff views consume it; future state-machine-iterated prompts get a single canonical implementation harder to misuse than reinventing the ref-key gate inline. Six unit tests in `tests/component/use-once-per-key.test.tsx` pin the contract: first-call-true, second-call-same-key-false, key-change-true-again, key-back-to-prior-key (treats as fresh — the hook only remembers the last key, matching forward-only iteration semantics), null-key gotcha.
- **Two component regression tests** in `tests/component/AdoptDiffView.test.tsx` + `tests/component/DiffView.test.tsx` re-render the same root with a new path (the iteration shape both parents use in production) and assert the second file's `onChoice` fires. The existing same-mount double-fire tests still pass.
- **Audit across every menu component** for the same pattern. `RemovedFilesReview` and `NewValuesPrompt` iterate internally with `setIndex` + `key={iterationKey}` on the inner widget, and their dedup ref only flips on the terminal item — correct as-is (Pattern A in the new doc). `CollisionReview`, `ExistingInstallGate`, `NewModulesReview`, and the three `Select`s inside `InstallWizard` (HeaderStep / ComputedPreview / ConfirmStep) are single-mount per phase and not vulnerable. State-machine hooks (`use-{install,update,adopt}-machine.ts`) hold per-command refs (phase, backupDir, addedPaths, hookAbort) that fire once per command invocation — different surface, also not vulnerable. No other instances of the bug pattern across `source/`.
- **`docs/COMPONENTS.md`** (new) — codifies the two valid shapes (Pattern A: internal iteration, Pattern B: state-machine iteration with `useOncePerKey`), the anti-pattern that broke `AdoptDiffView` and `DiffView`, the single-mount cases where boolean `useRef` is correct, and the `rerender()` regression-test convention with a canonical example.
- **`CLAUDE.md`** updates: source-of-truth table grows a `docs/COMPONENTS.md` row; project tree adds `source/components/use-once-per-key.ts` and lists the previously-omitted `docs/SHARD-LAYOUT.md` / `AUTHORING.md` / `ERRORS.md` / `OPERATIONS.md` (a separate stale-tree finding from the audit, fixed in scope); §Testing gets a binding rule that iterated-component regression tests via `rerender()` are mandatory.
- **Tests: 868 → 876 (+8)**.

## [0.1.1] - 2026-04-26

### Fixed (wizard select stuck on Enter when default = first option)

Closes [#103](https://github.com/breferrari/shardmind/issues/103). First entry on the v0.1.x stabilization track. Surfaced during a real `shardmind adopt` against `breferrari/obsidian-mind`, where the schema default for `vault_purpose` (`engineering`) matched the first option — pressing Enter on the focused option did nothing and the wizard froze. Blocks any shard whose schema has a `select` value with `default = first option`.

- **`source/components/ValueInput.tsx`** — the `select` branch reorders options so the option matching `initial` (`initialValue ?? def.default`) sits at index 0, and drops the `defaultValue` prop. Upstream cause: `@inkjs/ui`'s `use-select-state.js` seeds `previousValue === value === defaultValue` while `focusedValue` initializes to the first option; on Enter the change-fire guard (`previousValue !== value`) then fails when `defaultValue` equals the first option, so `onChange` never fires. Dropping `defaultValue` makes `previousValue` start undefined; the reorder keeps the cursor visually pre-positioned on the default.
- **Adversarial cases covered** — `tests/component/ValueInput.test.tsx` adds 6 cases: default = first option (the regression); default = middle option (reorder must surface non-first defaults at the cursor); single-option select with the bug pattern; no default + no initialValue; back-navigation prefill (`initialValue ≠ default`); `initialValue` not in options (graceful fallback to the first option). The existing `arrow + Enter` test stays. The diff also folds the repeated `await waitFor(() => onSubmit.mock.calls.length > 0 ? 'ok' : '', …)` idiom into a `waitForCall(fn)` helper in `tests/component/helpers.ts`.
- **No `source/core/` change.** Component-layer fix; no engine surface, no new error code, no new module, no spec or architecture-doc edits.
- **Tests: 862 → 868 (+6)**.

### Added (v6 layout: `shardmind adopt` command)

Spec: [`docs/SHARD-LAYOUT.md §Adopt semantics`](docs/SHARD-LAYOUT.md). The retrofit path for users who cloned a shard repo before shardmind support existed (e.g. obsidian-mind v5.1 clones). - **Spec clarification, pre-impl** (`docs/SHARD-LAYOUT.md §Adopt semantics`). The original 6-step list read as if walk preceded wizard, but `.njk` templates can't be classified without collected values. Reordered the flow to fetch → wizard → classify → per-file prompts → apply → hook → re-hash, mirroring how `runUpdate` already structures phases. Added: explicit pre-condition list (`.shardmind/state.json` absent → `ADOPT_EXISTING_INSTALL`; `shard-values.yaml` absent → `VALUES_FILE_COLLISION`); definition of "matches shard content exactly" for `.njk` files (post-frontmatter-normalization byte equality, the same equality `drift.ts` enforces); two-choice prompt rule (no third "leave untracked" branch — adopt is the entry point into management, divergence is handled later by the merge engine); volatile-template + excluded-module behavior. Single doc commit pinned upstream so review reads spec first.
- **`source/core/adopt-planner.ts`** — pure module + 19 unit cases + 2 fast-check properties. `classifyAdoption(input)` walks the shard via `resolveModules` (Tier 1 + `.shardmindignore` + symlink rejection apply transparently), renders each `.njk` template against the user's collected values (frontmatter goes through the same `parseYaml → stringifyYaml` normalization the renderer already applies during install), hashes the result, and compares against the user's vault. Output buckets: `matches` / `differs` / `shardOnly`. Volatile templates (`{# shardmind: volatile #}`) skip the differs prompt unconditionally (their rendered output is expected to vary across renders, so a content prompt would be meaningless). Module-excluded files are filtered before classification — adopt mirrors install's "module excluded → file not installed" rule, so user content at those paths stays user-content. User-only paths (in vault, not in shard) are deliberately not enumerated; classification is shard-source-driven, which keeps Tier 1 entries (`.git/`, `.obsidian/workspace.json`) and arbitrary user files out of scope and closes the symlink-escape vector at the user side. Read fan-out is capped at 32 via `mapConcurrent`, matching `drift.ts`'s budget.
- **`source/core/adopt-executor.ts`** — disk-mutating ops + 9 integration cases. `runAdopt(opts)` consumes the plan + per-`differs` resolution map and applies decisions:
  - `matches` → record managed FileState; no disk write.
  - `shard-only` → write shard bytes; record managed FileState; track in `addedPaths` so SIGINT rollback can erase only what this run created.
  - `differs + keep_mine` → record `ownership: 'modified'` with `rendered_hash = userHash`. Mirrors `update-executor`'s keep_mine-modified path so the next update's drift compares against the user's adopt-time bytes.
  - `differs + use_shard` → snapshot the user's existing bytes, then overwrite with shard bytes; record `ownership: 'managed'`.

  Pre-flight `assertAdoptable` rejects already-managed vaults and partial-state collisions BEFORE any disk mutation, so a deterministically-wrong adopt bails in milliseconds without leaving a backup directory behind. The `wx` flag on `writeValuesFile` is a belt-and-braces second defense against a values file appearing between guard and write. `rollbackAdopt(vaultRoot, backupDir, addedPaths)` restores snapshotted content, erases newly-introduced files, and drops `.shardmind/` + `shard-values.yaml`. Returns a per-file failure list rather than swallowing — telling the user "rolled back" while bytes remain stale is worse than telling them "rollback partially failed, here's what's still wrong".
- **2-way diff UI** (`source/components/AdoptDiffView.tsx`) — mirrors `DiffView`'s layout discipline (header line with N-of-M, hunk blocks with 3 lines of context above/below, color-coded mine vs shard, double-fire guard via `firedRef`) but with two choices (`keep_mine` / `use_shard`) instead of three. Hunks above the per-file cap of 8 collapse into a "…and N more diff regions" footer; the `Select` action stays available regardless. Binary fallback (planner-set `isBinary` flag) renders byte sizes only and never attempts a textual preview, but both choices remain so the user can still pick a side.

Spec: [`docs/SHARD-LAYOUT.md §Update semantics`](docs/SHARD-LAYOUT.md). Subsumes [#67](https://github.com/breferrari/shardmind/issues/67). Closes the `ARCHITECTURE.md §10.7` gap (current code hits `/releases/latest`, which 404s for beta-only repos). **Forward-only breaking** for the update path because the default-stable filter changes which tag is picked when a repo has both stable and prereleases — v0.1 has no published shards, so no migration needed.

- **`github:owner/repo#<ref>` syntax** (`source/core/registry.ts`) — branch / tag / commit-SHA addressing for direct-mode shards. Resolution is two API calls: `GET /commits/<encoded-ref>` returns the 40-char SHA; `HEAD /tarball/<sha>` verifies the tarball is fetchable. Refs are URL-encoded so `feature/foo` doesn't explode into two path segments. Refs are case-sensitive (GitHub's contract); owner / repo stay lowercase-only via the existing regex constraint. SHAs are lower-cased on the way out so `state.resolvedSha` is canonicalized regardless of upstream casing. The regex makes `@<version>` and `#<ref>` mutually exclusive at parse time — `o/r@v#ref` and `o/r#ref@v` both fail with `REGISTRY_INVALID_REF` instead of arbitrary precedence. Registry-mode refs (`o/r#main` without `github:`) are rejected — the index has no per-branch metadata, and committing to the direct flow's update semantics is part of the ref-install opt-in.
- **`/releases?per_page=100` filtered listing** replaces `/releases/latest` (`source/core/registry.ts::fetchLatestRelease`). Default policy returns the newest `prerelease: false` entry; `includePrerelease: true` widens. Empty filtered list throws `NO_RELEASES_PUBLISHED` with a hint that disambiguates "repo has zero releases" from "repo has only prereleases" (the latter points at `--include-prerelease`). 404 on `/releases` maps to `SHARD_NOT_FOUND` (the repo itself doesn't exist), not `NO_RELEASES_PUBLISHED` (which is reserved for "repo exists but has zero matching releases"). Malformed entries (missing `tag_name`, non-boolean `prerelease`) are silently skipped — one bad entry can't take down an otherwise-resolvable list.
- **`state.ref` + `state.resolvedSha`** (`source/runtime/types.ts`, `source/core/install-executor.ts`, `source/core/update-executor.ts`) — both optional. `ref` records the user-passed ref name; `resolvedSha` records the 40-char commit hex. Spread-on-presence keeps tag-install JSON tidy: no `null` keys for absent fields, so pre-#76 readers stay forward-compatible. `state.version` keeps tracking `manifest.version` regardless of how the shard was addressed, so semver-aware migrations work for ref installs too.
- **`update --release <tag>` + `update --include-prerelease`** (`source/commands/update.tsx`, `source/commands/hooks/use-update-machine.ts`). Three resolution policies: latest-stable (default), pinned-release (`--release`), widened (`--include-prerelease`). Mutual-exclusion guard rejects three combinations as `UPDATE_FLAG_CONFLICT`: `--release` + `--include-prerelease`; ref install + `--release`; ref install + `--include-prerelease`. Each rejection points at the unambiguous remediation in the hint. Ref installs always re-resolve via `/commits/<ref>` (not `/releases`); up-to-date short-circuit becomes SHA equality. Named `--release` rather than `--version` because Pastel reserves the program-level `--version` flag for printing the package version — a per-command `--version` would silently collide. The spec wording ("pin to a specific tag") stays semantically intact.
- **`primeLatestVersion` gate** — the update-check cache is "latest stable" by definition; the cache primer fires only when the run resolved through that policy. `--release`, `--include-prerelease`, and ref installs all skip the prime so the next `shardmind` (status) invocation isn't told a prerelease / pinned tag / SHA-derived version is the latest stable.
- **`resolveRefForUpdate` hint rewriting** preserves the registry-emitted hint when it mentions `--include-prerelease` (the beta-only sub-case is identical advice for the update audience), and rewrites the empty-list case for the update audience (drops "publish a release", suggests reinstalling).

### Added (post-install / post-update hook runtime)

Closes [#30](https://github.com/breferrari/shardmind/issues/30). `shard.yaml` `hooks.post-install` / `hooks.post-update` declarations are now actually executed at the end of install / update; previously the engine only *detected* the file and deferred execution (Summary rendered a yellow "not executed" notice). Unblocks Milestone 5 / [#14](https://github.com/breferrari/shardmind/issues/14) — obsidian-mind v6 ships a real `post-install.ts` that does first-run `git init` + scaffolding.

- **`source/core/hook.ts` — `executeHook`**: subprocess-backed runtime. Spawns `node --import <tsx loader> dist/internal/hook-runner.js <hookPath> <ctxFile>` with `cwd: ctx.vaultRoot` and a JSON-serialized `HookContext` delivered via a 0o600 tempfile (env-var would truncate over the 32 KB Windows cap; stdin would entangle with `source/core/cancellation.ts`). Captures stdout + stderr separately, caps each stream at 256 KB with a `[… truncated, N bytes discarded]` marker, and forwards chunks live through `onStdout` / `onStderr` callbacks for the streaming UI. Never throws — every failure mode (tsx missing, spawn error, timeout, abort, non-zero exit, hook throw) maps to a `HookResult` variant the caller surfaces in the summary.
- **`source/internal/hook-runner.ts`** (new): tiny ESM subprocess entry point. Reads argv, `pathToFileURL`-imports the hook, awaits `mod.default(ctx)`, exits 0/1. Zero Ink / React / Pastel imports — cold-start path. Emitted via a fourth tsup entry block at `dist/internal/hook-runner.js` and exposed via a new `./internal/hook-runner` subpath in `package.json`'s `exports` so `require.resolve` finds it regardless of how the package was installed.
- **Per-shard timeout override**: `shard.yaml` `hooks.timeout_ms` (integer, validated 1_000..600_000). Default `DEFAULT_HOOK_TIMEOUT_MS = 30_000` when absent. Enforcement is soft-SIGTERM + 2s grace + SIGKILL. Timeouts surface as `HookResult.failed` with a "timed out after Ns" message and any captured output preserved. `source/core/manifest.ts` + `source/runtime/types.ts`.
- **AbortSignal cancellation**: command machines thread an `AbortController` into `executeHook`. The `useSigintRollback` cleanup fires `.abort()` on every Ctrl+C so the child dies cleanly (the parent's existing SIGINT handler still wins; this is layered on top). Parent SIGINT during the hook phase does NOT trigger rollback — state.json is already on disk, per the Helm-style non-fatal contract (ARCHITECTURE §9.3). Result-decision order inside `executeHook` is cancel > timeout > spawn-error > exit so the user-facing message names the root cause rather than the symptom.
- **Streaming UI — new `running-hook` phase** in both `useInstallMachine` and `useUpdateMachine`. Carries a live output buffer capped at 64 KB (tighter than the core cap — this buffer lives in React state and re-renders per chunk). Rendered by the new `source/components/HookProgress.tsx` — spinner + stage-aware heading + tail-of-last-12-lines. Split on both LF and CRLF so Windows-authored hooks tail cleanly.

### Fixed (harden pass)

- **Binary assets no longer corrupt on `shardmind update`.** `detectDrift` was reading every tracked file as UTF-8 and hashing the decoded string, while install-executor hashed raw bytes. ### Fixed (Copilot review — 3 real regressions caught after rounds 1-7)

- **`shardmind` status command** — the last interactive command in v0.1. Reading a vault answers "what shard, what version, what drifted, is there an update?" in under a second. `shardmind` (no args) renders the quick view: identity line, install age, managed-file count, modified-by-you count, and a single explicit update line (`✓ Up to date` / `⬆ v4.0.0 available` / dim "unknown" with reason). `shardmind --verbose` appends five diagnostic sections: Values (valid count + any invalid keys), Modules (included / excluded), Files (every drift bucket with capped path lists), Frontmatter (per-file missing-key rows from `validateFrontmatter`), and Environment (Node.js version + Obsidian CLI PATH detection). Closes #13.
  - `source/core/status.ts` — pure aggregator. Loads state, cached manifest (fallback-synthesized from `state.shard` if the cached file is gone), cached schema, drift, update-check, and values in parallel. Verbose-only sections (per-modified-file `+N/−M` diff, frontmatter lint, environment probe) fan out via `Promise.all` — they're short-circuited in the common case so quick status stays fast. For each modified file the verbose pass re-renders the cached template against current values and runs a 2-way line diff (`diff.diffLines`) with CRLF + UTF-8-BOM normalization; a pure-whitespace change renders as `(whitespace-only)` rather than a confusing `+0/−0`. Capped at 20 files via `MAX_PATHS_PER_BUCKET` and 8-way concurrency via `MODIFIED_DIFF_CONCURRENCY`. Section-level failures degrade to `StatusWarning` entries instead of throwing; status never stack-traces on a read-only command. Path lists in every drift bucket cap at 20 with a `truncated` flag so vaults with thousands of orphans still render.
  - `source/core/update-check.ts` — **closes #51 (the 24h update-check cache)** ahead of v0.2 because status's UX depends on it. Cache file `.shardmind/update-check.json`, 24h TTL, atomic `writeFile → rename` so a crashed writer can't leave a half-written body visible to a reader. Corrupt JSON is deleted on sight and treated as absent. Source-mismatched caches invalidate automatically. Network failures with a prior cache entry return it as `stale` (still shown with a `(cached)` suffix); no cache + no network degrades to `unknown` rather than throwing. A 4-second fetch timeout prevents a hanging TCP connection from blocking status.
  - `source/core/registry.ts` grows `fetchLatestVersion(source)` — the lean "latest tag" lookup without the HEAD verification `resolve()` does (only needed when you actually want the tarball URL).
  - `source/commands/index.tsx` replaces the placeholder with a thin dispatcher on top of `useStatusReport`. Phases: `booting → loading → (not-in-vault | ready | error)`. No `CommandFrame` — that exists to host a dry-run banner + keyboard legend, neither of which applies to a read-only report; including them would suggest affordances that don't exist.
  - `source/commands/hooks/use-status-report.ts` — async loader with the five-phase discriminated-union pattern used by install/update. Defers the Ink exit by 50 ms so the final frame flushes before stdout tear-down.
  - `source/components/StatusView.tsx` + `source/components/VerboseView.tsx` — presentation-only, consume `StatusReport`, zero I/O. `StatusView` renders an explicit update line (never silently succeeds) so the user always knows the check ran. `VerboseView` extends it with the five diagnostic sections and respects the truncation flags the builder emits.
  - `source/commands/hooks/use-update-machine.ts` — after `resolve()` succeeds, fire-and-forget `primeLatestVersion(vaultRoot, state.source, resolved.version)` so the next `shardmind` run answers instantly. Swallows errors so a cache write failure can't fail an update.
  - Two new typed error codes: `UPDATE_CHECK_FAILED` (internal — surfaced when the 4-second fetch budget expires) and `UPDATE_CHECK_CACHE_CORRUPT` (surfaced as a verbose-only info warning whenever `readCache` detects and auto-heals a corrupt or directory-typed cache entry). `readCache` now returns `{ cache, corruptHealed }` and `UpdateCheckResult` carries an optional `cacheHealed` flag so the signal propagates cleanly from the cache layer through `getLatestVersion` to the status report. `UpdateStatus.reason` distinguishes `'no-network'` (real fetch failure) from `'cache-miss'` (caller skipped the lookup) from `'unsupported-source'` (non-`github:` scheme) — each renders a distinct message. New `StatusReport`, `UpdateStatus`, `StatusWarning`, `StatusDriftSummary`, `StatusModuleSummary`, `StatusValuesSummary`, `StatusFrontmatterSummary`, `StatusEnvironmentReport`, `StatusModifiedChanges` types exported from `shardmind/runtime`.
  - Flags: `--verbose` (populate every diagnostic section). No `--dry-run` because status is read-only.
  - **Deviations from the spec** (`docs/ARCHITECTURE.md §10.3`) documented explicitly: per-file *natural-language* flavor text like `"you added a custom section"` would require semantic diff we don't have, so verbose mode renders `path — +N/−M` (or `(whitespace-only)` / `(diff unavailable)`) instead; shard-specific environment checks (e.g., "QMD not installed") require a `status` hook contract that doesn't exist yet (hooks are post-install/post-update only). Both are documented in the verbose section header in ARCHITECTURE.md.

- **`shardmind update` command** — the feature the three-way merge engine was built to serve. Full pipeline: fetch new shard → apply schema migrations → prompt only for newly required values → offer newly optional modules → decide per-file fate for modified files the new shard no longer ships → three-way merge every modified file in parallel → resolve conflicts via `DiffView` → snapshot-and-write → post-update hook → summary. Any failure between snapshot and state-write walks the snapshot back; the vault is indistinguishable from pre-update.
  - `source/core/migrator.ts` — `applyMigrations(values, current, target, migrations)` transforms `shard-values.yaml` across versions. Filter rule is `currentVersion < from_version ≤ targetVersion` (idempotent: re-running an upgrade at the same target version picks up nothing). Four change types (`rename` / `added` / `removed` / `type_changed`); `rename` refuses to clobber an existing target value — warns and keeps both keys instead of destroying user data.
  - `source/core/update-planner.ts` — pure planner. Inputs grouped as `{ vault, values, newShard, removedFileDecisions }` so call sites can't accidentally mix fields from two different shards. Emits `UpdatePlan` with 10 action variants (`overwrite / auto_merge / conflict / noop / skip_volatile / add / restore_missing / delete / keep_as_user`) plus a `pendingConflicts` queue for the state machine to drive. Per-modified-file merges run in parallel via `mapConcurrent(drift.modified, 16, …)`. `theirsHash` is captured at plan time and threaded through to the executor so `keep_mine`/`skip` resolutions don't need to re-read + re-hash the file.
  - `source/core/update-executor.ts` — applies the plan with snapshot-based rollback. Backup directory allocation is collision-safe under same-millisecond concurrent updates (ISO timestamp retains ms, then a numeric suffix is probed if EEXIST). Snapshot copies the plan's touched files plus `.shardmind/state.json`, cached manifest + schema + templates in parallel (`SNAPSHOT_CONCURRENCY=16`). Write pass runs before delete pass so a rename-style move (delete + add at a different path) can't clobber the incoming file.
  - `source/core/values-io.ts` — one YAML reader for both install's `--values` prefill and update's canonical `shard-values.yaml` load. Install filters unknown keys against the schema; update keeps everything so migrations can handle the shape change. Same read → parse → type-check code path.
  - `source/commands/update.tsx` + `source/commands/hooks/use-update-machine.ts` — state machine with phases `booting → loading → (up-to-date | prompt-new-values → prompt-new-modules → prompt-removed-files → resolving-conflicts) → writing → summary`. The `no-install` case throws `UPDATE_NO_INSTALL` (exit 1) — see the E2E Changed entry above. Mirrors the `useInstallMachine` pattern.
  - `source/commands/hooks/shared.ts` — `summarizeHook` and `useSigintRollback` shared across install + update. SIGINT always runs the tempdir cleanup (plugging a leak where cancelling during the prompt/wizard phase left the extracted shard on disk).

  - Six new error codes typed in `source/runtime/errors.ts`: `UPDATE_NO_INSTALL`, `UPDATE_SOURCE_MISMATCH`, `UPDATE_CACHE_MISSING`, `UPDATE_WRITE_FAILED`, `MIGRATION_INVALID_VERSION`, `MIGRATION_TRANSFORM_FAILED`.
  - Flags: `--yes` (auto-accept every prompt; auto-keep conflicts; include every new optional module), `--verbose` (per-file action history during write), `--dry-run` (plan + summarize without touching the vault or allocating a backup).

- **Three-way merge engine** (`source/core/drift.ts`, `source/core/differ.ts`). Solves "propagate template updates to existing vaults" — the same problem Backstage (Spotify) has had open for 3+ years and that cruft/create-react-app/Yeoman never solved. Uses `node-diff3`'s Khanna–Myers algorithm via `diff3MergeRegions` for accurate stats.
  - `detectDrift()` classifies every tracked file into `managed / modified / volatile / missing / orphaned` by sha256-comparing disk content against `state.json.rendered_hash`. Orphan detection is non-recursive: parent directories of tracked files become tracked dirs; files in a tracked dir not in `state.files` are orphans. Engine scaffolding (`shard-values.yaml`, `.shardmind/`) and third-party metadata (`.git/`, `.obsidian/`) are excluded.
  - `computeMergeAction()` returns `skip | overwrite | auto_merge | conflict` — skip when `base === ours`, overwrite for managed files, three-way merge for modified files. Conflict markers use git vocabulary: `<<<<<<< yours` / `=======` / `>>>>>>> shard update`.
  - `threeWayMerge()` is a pure primitive — line-based, CRLF-tolerant on input, LF on output.

### Changed

- `InstallProgress` renamed to `CommandProgress`; both install and update commands render it. Callers should import from `source/components/CommandProgress.js`.
- `mapConcurrent` moved from `source/core/drift.ts` to `source/core/fs-utils.ts` so planner and executor can share the bounded-concurrency primitive.
- `CommandFrame` extracted — was `RootFrame` in `install.tsx` and `Frame` in `update.tsx`. Single component owns the dry-run banner + keyboard legend.
- `ShardMindError.code` typed as `ErrorCode` instead of free-form `string` (compile-time check of every call site).
- `detectDrift` now runs per-file reads in parallel via `Promise.all`, and runs the orphan scan in parallel with the classification.
- `source/runtime/errno.ts` centralizes `errnoCode` / `isEnoent`; 8 copies of the `err instanceof Error && 'code' in err ? ...` pattern collapsed.

### Fixed

- `rename` migration no longer silently overwrites an existing value at the target key. If both the old and new keys have values, the migrator warns and keeps both (the user picks which to drop manually). Previously this was a silent data-loss bug found by the adversarial audit.
- `createBackupDir` allocates distinct backup directories even when called twice in the same millisecond. Previous timestamp stripped the fractional seconds, so two near-simultaneous updates could share a backup dir and clobber each other's rollback snapshots.
- `keep_as_user` decisions now untrack the file from `state.files` on apply. The file stays on disk (as the user asked), but the engine no longer considers it managed on future updates — matches the UI's "Keep my edits (untrack)" label.
- `DiffView` splits on `/\r?\n/` everywhere to match `differ.ts`'s canonical line splitter. Previously, CRLF content left `\r` in the output strings and Ink's renderer treated them as cursor-moving carriage returns, corrupting the terminal display.
- `conflictFromDirect` now receives the new-shard tempdir and produces a cache-relative `templateKey` instead of an absolute path. The fallback fires only on corrupt / missing cached templates, but when it did fire the state it wrote couldn't be re-used by subsequent updates.
- SIGINT now runs the shard tempdir cleanup regardless of which phase was active. Previously, cancelling with Ctrl-C during the wizard/prompt phases left the extracted tarball in the OS temp directory until the next reboot.
- CRLF on Windows-saved user files no longer produces spurious conflicts against LF-rendered base/ours.
