### Changed

- **`shardmind update` on a no-install directory now throws a typed `UPDATE_NO_INSTALL` error (exit 1)** instead of rendering a friendly `no-install` phase (exit 0). CLI scripting / CI integrations can now detect the no-install case ‚Äî the existing "warm" wording is preserved as the error's hint. Spec parity: `docs/IMPLEMENTATION.md ¬ß7` had listed this code as thrown by `use-update-machine`; until now the declaration was dead.
- **`shardmind update` with a malformed `state.source` now throws `UPDATE_SOURCE_MISMATCH` (exit 1)** instead of surfacing the generic `REGISTRY_INVALID_REF` hint that pointed the user to fix their CLI input (unhelpful ‚Äî the bad ref came from `state.json`, not the command line). The new hint points at the actual repair site.
- **`shardmind install` + `shardmind update` set `process.exitCode = 1` on error-phase transitions** so scripting / CI can detect failure. `cancelled`, `summary`, `up-to-date`, and every success path keep exit 0. Status stays at exit 0 on every phase by design ‚Äî per `ARCHITECTURE ¬ß10.2` it's an ambient read-only surface and must never stack-trace the shell (the typed code still surfaces to stdout).
- **`backupCollisions` in `source/core/install-executor.ts` is now transactional.** Previously, if one rename failed mid-loop, the loop threw `BACKUP_FAILED` but the renames done earlier were left in their `.shardmind-backup-*` locations ‚Äî the vault ended with user content silently stashed at a backup path the error didn't mention. The function now walks back every successful rename before throwing; if the restore-walk itself fails, the orphan path is named explicitly in the error's hint so manual recovery is possible.
- **CI reorders `npm run build` before `npm test`** (`.github/workflows/ci.yml`) so the E2E suite has `dist/cli.js` to spawn. The local-dev flow is covered by `tests/e2e/helpers/build-once.ts`; CI is explicit for hermetic installs.

### Fixed

- `shardmind install` and `shardmind update` surface a proper non-zero exit code on error. Previously every error path exited 0, making the CLI unusable for scripting (an invalid shard ref or missing values file returned "success"). Now `error` phase ‚Üí exit 1; `summary` / `cancelled` / `up-to-date` ‚Üí exit 0. Status (`shardmind`) intentionally keeps exit 0 on every phase ‚Äî it's an ambient read-only surface, per `ARCHITECTURE ¬ß10.2` it must never stack-trace the shell.
- Update-path error hints no longer misdirect users to fix their command line. `resolveRefForUpdate` (extracted from `use-update-machine.ts`) now rewrites three codes for the update audience: `REGISTRY_INVALID_REF` ‚Üí `UPDATE_SOURCE_MISMATCH` with the malformed value embedded; `SHARD_NOT_FOUND` and `VERSION_NOT_FOUND` keep their codes but gain update-specific hints pointing at `state.source` and the real remediation (reinstall / retry / check upstream) instead of the install-path "check spelling" / "pick an available version" phrasings ‚Äî which apply to CLI input the user never typed during update. 7 new unit tests in `tests/unit/update-machine.test.ts` pin every branch.
- Windows now exercises the SIGINT rollback path in CI. A stdin-ETX bridge (`source/core/cancellation.ts`) re-emits SIGINT inside the child's own process when Node's Windows-quirk `TerminateProcess` would otherwise skip every registered handler; two previously `it.skipIf(win32)` E2E scenarios now run unconditionally.
- `backupCollisions` walks back partial work on failure ‚Äî see Changed block above; previously it silently left user content at backup paths when a later rename failed. This was not caught by any existing test and only surfaced during the E2E audit.

### Fixed

- `rename` migration no longer silently overwrites an existing value at the target key. If both the old and new keys have values, the migrator warns and keeps both (the user picks which to drop manually). Previously this was a silent data-loss bug found by the adversarial audit.
- `createBackupDir` allocates distinct backup directories even when called twice in the same millisecond. Previous timestamp stripped the fractional seconds, so two near-simultaneous updates could share a backup dir and clobber each other's rollback snapshots.
- `keep_as_user` decisions now untrack the file from `state.files` on apply. The file stays on disk (as the user asked), but the engine no longer considers it managed on future updates ‚Äî matches the UI's "Keep my edits (untrack)" label.
- `DiffView` splits on `/\r?\n/` everywhere to match `differ.ts`'s canonical line splitter. Previously, CRLF content left `\r` in the output strings and Ink's renderer treated them as cursor-moving carriage returns, corrupting the terminal display.
- `conflictFromDirect` now receives the new-shard tempdir and produces a cache-relative `templateKey` instead of an absolute path. The fallback fires only on corrupt / missing cached templates, but when it did fire the state it wrote couldn't be re-used by subsequent updates.
- SIGINT now runs the shard tempdir cleanup regardless of which phase was active. Previously, cancelling with Ctrl-C during the wizard/prompt phases left the extracted tarball in the OS temp directory until the next reboot.
- CRLF on Windows-saved user files no longer produces spurious conflicts against LF-rendered base/ours.
